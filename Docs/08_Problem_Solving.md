# Problem Solving & Technical Decisions

## 1. 개요

이 문서는 PoF 프로젝트를 진행하면서 실제로 겪었던 문제와  
그 문제를 어떤 방식으로 분석하고 해결했는지를 정리한 문서입니다.

이 프로젝트를 진행하면서 중요하게 본 것은 단순히 기능을 추가하는 것이 아니라,

- 데이터 흐름이 끊기지 않는지
- 저장 / 불러오기 이후에도 상태가 유지되는지
- 탐험 / 전투 / 보상 / 성장 시스템이 자연스럽게 연결되는지

를 확인하고, 문제가 생겼을 때 구조적으로 개선하는 것이었습니다.

---

## 2. Quest Continue 초기화 문제

### 문제 상황

이어하기(Continue) 후 탐험 씬으로 복귀했을 때 다음 문제가 발생했습니다.

- 진행 중이던 퀘스트의 수치가 0으로 초기화됨
- 예를 들어 광석을 20개 모으는 퀘스트에서 14개를 모은 상태로 저장했는데,
  Continue 이후 다시 0개로 보이는 문제 발생
- 진행 중이던 퀘스트가 현재 단계가 아니라 1번 퀘스트로 되돌아가는 현상 발생

즉, 플레이어 입장에서는 분명 저장했는데  
퀘스트 진행 상태가 이어지지 않는 문제가 있었습니다.

---

### 원인 분석

이 문제는 단순 UI 문제가 아니라  
퀘스트 상태를 저장하고 복원하는 구조가 충분히 분리되어 있지 않았기 때문에 발생했습니다.

주요 원인은 다음과 같았습니다.

#### 1) 퀘스트를 index 기반으로 다루던 흐름
퀘스트를 “몇 번째 퀘스트인지” 기준으로만 다루면  
데이터 추가/수정 또는 로딩 순서 차이 때문에 잘못된 퀘스트가 복원될 수 있습니다.

#### 2) 진행 수치가 정적 데이터와 혼합되어 있던 문제
`QuestData`는 설계용 데이터인데,  
실제 진행 상태까지 같은 흐름으로 취급하면 저장/복원이 꼬일 가능성이 높습니다.

#### 3) Continue 시 Runtime Progress 재구성 부족
저장 파일에서 퀘스트 ID를 읽은 뒤,  
그 ID에 대응되는 `QuestData`를 다시 찾고  
`QuestRuntimeProgress`를 다시 만들어야 하는데 이 연결이 약했습니다.

---

### 해결 방식

#### 1) questId 기반 복원 구조로 변경
퀘스트를 인덱스가 아니라 `questId` 기준으로 저장하고 복원하도록 변경했습니다.

예를 들어:

- `Q_MAIN_001`
- `Q_GATHER_002`

같은 식으로 고유 ID를 기준으로 현재 퀘스트를 다시 찾게 구성했습니다.

이렇게 하면 퀘스트 배열 순서가 바뀌더라도  
현재 퀘스트를 비교적 안정적으로 복원할 수 있습니다.

---

#### 2) 진행 상태(Runtime) 분리
정적 데이터와 동적 상태를 분리했습니다.

- `QuestData` → 퀘스트 설계 정보
- `QuestRuntimeProgress` → 현재 플레이 중 진행 수치 / 완료 여부 / 보상 수령 여부

이렇게 분리하면  
설계 데이터는 그대로 유지하면서,  
실제 진행 상태는 저장/로드 전용 구조로 다룰 수 있습니다.

---

#### 3) SaveData에 퀘스트 상태 명시 저장
퀘스트 복원을 위해 아래 필드를 명시적으로 저장했습니다.

- `currentQuestId`
- `currentQuestValue`
- `currentQuestCompleted`
- `currentQuestRewardClaimed`
- `completedQuestIds`

즉, “현재 어떤 퀘스트를 하고 있는지”와  
“그 퀘스트가 어디까지 진행됐는지”를 저장 파일에 직접 남기도록 했습니다.

---

#### 4) 복원 실패 감지 구조 추가
Continue 시 `questId`는 있는데  
실제 `GameDataRegistry`에서 해당 `QuestData`를 찾지 못하는 경우를 감지하도록 했습니다.

이렇게 하면 잘못된 데이터 상태에서 무조건 이어하기를 강행하지 않고,  
복원 실패를 명확히 인식하고 대응할 수 있습니다.

---

### 결과

이 개선 이후:

- Continue 후에도 퀘스트 진행 수치가 유지됨
- 현재 진행 중인 퀘스트가 정확히 복원됨
- 퀘스트 완료 / 보상 수령 상태도 분리되어 안정적으로 유지됨

즉, 퀘스트 시스템은 단순 표시용이 아니라  
저장/불러오기까지 고려한 구조로 안정화되었습니다.

---

## 3. 전투 ↔ 탐험 상태 동기화 문제

### 문제 상황

전투가 끝난 뒤 탐험 씬으로 돌아왔을 때:

- 전투 중 감소한 HP가 탐험 쪽에 반영되지 않음
- 전투에서 사용한 SP 상태가 이어지지 않음
- 전투 결과가 실제 파티 상태와 분리되는 현상 발생

즉, 전투에서는 분명 변화가 일어났는데  
탐험으로 복귀하면 그 결과가 사라지는 문제가 있었습니다.

---

### 원인 분석

전투에서는 `BattleActorRuntime`을 사용하고,  
탐험/전역 상태는 `CharacterRuntime`을 사용합니다.

이 구조 자체는 맞지만,  
전투 종료 후 이 결과를 `GameContext.party` 쪽으로 다시 반영하는 단계가 없거나 약하면  
전투 결과가 실제 게임 상태에 남지 않습니다.

즉, 문제의 핵심은:

- 전투 Runtime과 전역 Runtime은 분리되어 있었고
- 그 둘을 연결하는 Sync 단계가 부족했다

는 점이었습니다.

---

### 해결 방식

#### `SyncBattlePartyStateToGameContext()` 추가 / 강화
전투 종료 시점에  
전투 중 변경된 HP / MaxHP / SP를  
`GameContext.party`의 `CharacterRuntime`에 다시 반영하는 함수를 명확히 두었습니다.

핵심은 다음과 같습니다.

- 현재 Battle 쪽 actor와 원본 party index를 매핑
- 전투 종료 시 그 매핑 기준으로 상태 반영
- 탐험 복귀 전에 반드시 Sync 수행

---

### 결과

이 개선 이후:

- 전투 후 줄어든 HP가 탐험에서도 유지됨
- 전투 자원 상태가 실제 플레이 흐름에 반영됨
- 탐험 ↔ 전투 ↔ 탐험 루프가 끊기지 않게 됨

즉, 전투는 별도 씬이지만  
결과는 GameContext를 통해 일관되게 이어지는 구조를 확보했습니다.

---

## 4. Secret Art 중복 적용 문제

### 문제 상황

탐험에서 Secret Art를 준비한 뒤 전투에 들어가면  
전투 시작 효과가 한 번만 적용되어야 했지만,  
특정 상황에서는 중복 적용 가능성이 있었습니다.

예를 들어:

- 전투 시작 시 파티 회복이 두 번 들어가거나
- 시작 버프가 의도보다 많이 적용될 가능성

이 구조는 전투 밸런스를 무너뜨릴 수 있기 때문에  
반드시 막아야 하는 문제였습니다.

---

### 원인 분석

문제의 핵심은 “전투 시작 처리”가 여러 번 불릴 가능성이 있다는 점입니다.

- 전투 초기화 루틴이 복잡해질수록
- 전투 시작 훅이 여러 경로에서 불릴수록
- Secret Art 적용 코드가 방어적으로 작성되지 않으면

중복 적용 위험이 생깁니다.

또한 적용 후 `secretArtReady` 상태를 바로 해제하지 않으면  
다음 흐름에서 다시 참조될 수 있습니다.

---

### 해결 방식

#### 1) 1회 적용 플래그 추가
전투 시작 시 Secret Art를 적용했는지 여부를 확인하는 플래그를 두었습니다.

예:
- `secretArtApplied`

이 플래그가 true이면  
같은 전투에서 다시 적용하지 않도록 막았습니다.

---

#### 2) 적용 후 ready 상태 해제
Secret Art는 “탐험에서 준비하고, 전투 시작 시 1회 적용되는 효과”이므로  
적용 직후 해당 캐릭터의 `secretArtReady`를 false로 변경했습니다.

즉:

- 탐험 준비 상태 → true
- 전투 시작 시 적용 → 효과 발동
- 적용 직후 → false

이 흐름을 명확히 했습니다.

---

### 결과

이 개선 이후:

- Secret Art 효과는 전투 시작 시 정확히 1회만 적용
- 캐릭터별 비술 효과가 의도한 정책대로 동작
- 탐험과 전투 간 상태 전이가 더 안정적이 됨

---

## 5. 애니메이션 이벤트와 실제 데미지 타이밍 불일치 문제

### 문제 상황

전투에서 공격 애니메이션이 재생되는 시점과  
실제 데미지가 적용되는 시점이 완전히 맞지 않는 문제가 있었습니다.

이 경우 다음 문제가 생길 수 있습니다.

- 칼이 닿기 전에 데미지가 먼저 들어감
- 적이 맞았는데 피격 판정이 늦게 뜸
- 애니메이션 이벤트가 누락되면 전투 흐름이 멈출 수 있음

즉, 연출과 판정이 분리되어야 했습니다.

---

### 원인 분석

애니메이션만 믿고 전투 판정을 처리하면  
다음 문제가 생길 수 있습니다.

- 이벤트 누락
- 클립 수정 시 타이밍 붕괴
- 특정 적/보스 패턴에서 예외 발생

특히 보스 패턴처럼 연출이 길거나 특수 이벤트가 있는 경우  
단순 고정 타이밍만으로는 대응이 어렵습니다.

---

### 해결 방식

#### 1) Animation Event 우선 구조
가능하면 애니메이션 이벤트를 기준으로  
실제 타격 판정을 실행하도록 했습니다.

---

#### 2) Fallback 타이밍 추가
이벤트가 누락되거나 실패했을 경우  
전투가 멈추지 않도록 fallback delay를 두었습니다.

즉:

- 이벤트가 오면 그 타이밍 사용
- 이벤트가 안 오면 delay 기반으로 안전 처리

이 구조로 바꿨습니다.

---

#### 3) 타이밍 변수 분리
전투 연출과 로직을 조정하기 쉽도록  
타이밍 관련 변수를 분리했습니다.

예:
- `hitTimingDelay`
- `afterAttackAnimHold`
- 보스 전용 hold 값들

---

### 결과

이 개선 이후:

- 전투 연출과 실제 판정의 일치도가 올라감
- 이벤트 누락 시에도 전투가 멈추지 않음
- 일반 적 / 보스 패턴 모두 안정적으로 처리 가능

---

## 6. Git 대용량 파일 문제

### 문제 상황

프로젝트를 GitHub에 올리는 과정에서  
push가 반복적으로 실패했습니다.

실제 문제는 다음과 같았습니다.

- 첫 push가 2GB 이상으로 커짐
- `HTTP 500`, `RPC failed` 오류 발생
- GitHub 100MB 제한 초과 파일 존재
- `Recordings/Movie_002.mp4` 같은 대용량 파일 때문에 업로드 거절

---

### 원인 분석

Unity 프로젝트 전체를 그대로 올리면  
다음 문제가 발생할 수 있습니다.

- `Library`, `Temp`, `Logs` 같은 캐시 폴더 포함
- 대용량 녹화 파일 포함
- 아트/오디오 원본까지 같이 포함
- 첫 push 용량이 지나치게 커짐

즉, “프로젝트 전체”와 “버전 관리할 핵심 파일”을 구분해야 했습니다.

---

### 해결 방식

#### 1) `.gitignore` 정리
Git으로 관리하지 않을 폴더를 명확히 제외했습니다.

예:
- `Library/`
- `Temp/`
- `Logs/`
- `.vs/`
- `.vscode/`
- `Recordings/`

---

#### 2) 대용량 에셋 폴더 제외
초기 GitHub 업로드에서는  
핵심 코드/씬/프리팹/ScriptableObject 중심으로만 관리하도록 했습니다.

예:
- `Assets/Game/05.Art/`
- `Assets/Game/07.Audio/`

를 초기 push 대상에서 제외

---

#### 3) 필요한 폴더만 선택적으로 add
처음에는 `git add .`가 아니라  
필요한 폴더만 직접 추가하는 방식으로 정리했습니다.

예:
- `Assets/Game/01.Scenes`
- `Assets/Game/02.Scripts`
- `Assets/Game/03.ScriptableObjects`
- `Assets/Game/04.Prefabs`
- `Packages`
- `ProjectSettings`

---

### 결과

이 개선 이후:

- GitHub push 성공
- 핵심 프로젝트 구조를 안전하게 버전 관리 가능
- 포트폴리오용 저장소로도 적절한 상태 확보

---

## 7. 정리 과정에서 중요하게 본 기술적 판단

이 프로젝트를 진행하면서 단순히 “돌아가게 만드는 것”보다  
다음 판단을 더 중요하게 보았습니다.

### 1) SSOT 유지
여러 시스템이 각자 상태를 따로 들고 있지 않고,  
최종 기준은 GameContext로 통일하려고 했습니다.

---

### 2) 정적 데이터 / 동적 상태 분리
`CharacterData`, `QuestData`, `ItemData` 같은 정적 데이터와  
실제 플레이 중 변하는 Runtime 상태를 분리했습니다.

---

### 3) 탐험 / 전투 / 보상 / 성장 연결
각 시스템을 따로 만드는 데서 끝내지 않고,  
실제 게임 루프로 자연스럽게 이어지도록 구성했습니다.

---

### 4) 실제 문제를 기준으로 구조 개선
추상적인 리팩토링보다  
Continue 오류, 상태 동기화, 중복 적용, 업로드 문제처럼  
실제로 발생한 문제를 기준으로 구조를 개선했습니다.

---

## 8. 개인적으로 가장 중요하게 본 부분

이 프로젝트에서 가장 중요하게 본 것은  
“기능 개수”가 아니라

👉 **데이터 흐름과 상태 일관성**

이었습니다.

특히 아래 세 가지를 가장 중요하게 다뤘습니다.

- Continue 후에도 상태가 정확히 복원되는지
- 전투 결과가 탐험으로 정확히 이어지는지
- 여러 시스템이 같은 기준 데이터를 바라보는지

이 부분을 기준으로 구조를 다듬으면서  
프로젝트 전체 안정성을 높이는 방향으로 개선했습니다.
